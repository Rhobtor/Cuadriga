

# ARGJ801 Setup Package

## Overview

El paquete `argj801_setup` centraliza la **configuración** y el **arranque** (launch) del
sistema ROS 2 del rover **ARGJ801**. Su función es:

- Cargar parámetros comunes desde `config/J8_params.yaml`.
- Lanzar nodos de control, planificación y sensorización bajo el namespace `/ARGJ801`.
- Orquestar transiciones de *Lifecycle* (configure/activate) con retardos para evitar
  carreras durante el arranque.

Este paquete no implementa lógica de control; sólo describe cómo se inicia el sistema.

## Installation

1. Clone the repository:
   ```sh
   git clone git@github.com:Robotics-Mechatronics-UMA/argj801_setup.git
   cd argj801_setup
   ```

2. Install dependencies:
   ```sh
   rosdep install --from-paths src --ignore-src -r -y
   ```

3. Build the package:
   ```sh
   colcon build
   ```

## Usage

### Launch the System

El launch principal es `launch/J8_launch.py`. Este fichero puede usarse tanto para
robot real como para simulación, habilitando/deshabilitando subconjuntos de nodos.

#### Parameters

- `robot`: Launch robot-specific nodes if `true` (default: `false`).
- `local_test`: Launch a PC-only validation setup without robot hardware, keeping FSM, path manager, teleoperation, GUI and the integrated Cuadriga actuator available (default: `false`).
- `simulator`: Launch simulator-specific nodes if `true` (default: `false`).
- `use_gui`: Launch the GUI node if `true` (default: `false`).
- `use_cuadriga_mock`: In `local_test`, start a fake Cuadriga backend that consumes `/Cuadriga/serial_write` and publishes simulated IMU/GPS/Odometry for debugging without hardware (default: `true`).

#### Examples

1. **Launching the Simulator without GUI:**

    ```sh
  ros2 launch argj801_setup J8_launch.py simulator:=true use_gui:=false
    ```

2. **Launching the Real Robot:**

    ```sh
  ros2 launch argj801_setup J8_launch.py robot:=true
    ```

3. **Launching the Simulator with GUI:**

    ```sh
  ros2 launch argj801_setup J8_launch.py simulator:=true use_gui:=true
    ```

4. **Launching a local PC test without robot hardware:**

    ```sh
  ros2 launch argj801_setup J8_launch.py local_test:=true use_gui:=true
    ```

    This mode keeps the FSM and the integrated Cuadriga actuator active, but skips robot-only nodes such as the Fixposition driver, `follow_zed_node` and the Android TCP server.

    The actuator exposes two command inputs:
    - `/Cuadriga/direct_cmd_vel`: direct velocity command, always available even without using the FSM
    - `/ARGJ801/cmd_vel`: command generated by the J8 FSM stack

    While direct commands are being received recently, they take priority over the FSM command channel. If direct commands stop, the actuator falls back automatically to the FSM channel.

5. **Launching a fake robot for Cuadriga debugging:**

    ```sh
  ros2 launch argj801_setup J8_launch.py local_test:=true use_gui:=false use_cuadriga_mock:=true
    ```

    This starts a mock backend that:
    - subscribes to `/Cuadriga/serial_write`
    - decodes the Cuadriga ASCII commands produced by the integrated `cuadriga_actuator_node`
    - publishes fake telemetry on `/imu_head/data`, `/fixposition/odometry_enu`, `/fixposition/navsatfix` and `/fixposition/navsatfix/vel`

    Useful setup with two computers on the same LAN:
    - PC A: launch the fake robot command above
    - PC B: run `j8_gui` or the full GUI container against the same ROS/DDS network

    This lets you debug FSM transitions, GUI telemetry, `/ARGJ801/cmd_vel`, `/Cuadriga/direct_cmd_vel` and the Cuadriga actuator path without the real robot.

### Docker execution

The workspace root `dockerfile` already installs dependencies and builds the full ROS 2 workspace, including `cuadriga_actuator`.

Build the image from the workspace root:

```sh
docker build -t cuadriga-j8 -f dockerfile .
```

Run a local integration test without hardware:

```sh
docker run --rm -it \
  --net=host \
  --ipc=host \
  cuadriga-j8 \
  ros2 launch argj801_setup J8_launch.py local_test:=true use_gui:=false use_cuadriga_mock:=true
```

Run on the real Cuadriga robot with serial access:

```sh
docker run --rm -it \
  --net=host \
  --ipc=host \
  --device=/dev/ttyUSB0:/dev/ttyUSB0 \
  cuadriga-j8 \
  ros2 launch argj801_setup J8_launch.py robot:=true use_cuadriga_backend:=true
```

To command the robot directly without entering any FSM mode:

```sh
ros2 topic pub /Cuadriga/direct_cmd_vel geometry_msgs/msg/Twist \
  '{linear: {x: 0.4}, angular: {z: 0.0}}'
```

If direct commands stop arriving, the actuator falls back automatically to the FSM command topic `/ARGJ801/cmd_vel`.

By using these parameters, you can easily switch between launching the system for the real robot or the simulator, and optionally include the GUI.
### Nodes

The `argj801_setup` package launchs the following nodes:

- `ctl_mission_node`
- `controller_node`
- `path_following_node`
- `teleoperation_node`
- `path_record_node`
- `ready_node`
- `estop_node`
- `back_home_node`
- `path_manager_node`
- `security_check_node`
- `mpc_node`
- `cuadriga_actuator_node`
- `joy_node`
- `android_server_node`
- `laser_segmentation`
- `static_transform_publisher`
- `gui_node`

When `use_cuadriga_backend:=false`, the legacy `argj801_ctrl_platform_node` can still be launched instead of the integrated Cuadriga actuator.

Each node is configured and launched with specific parameters to perform its designated function in the system.

### Lifecycle / secuencia de arranque

Muchos de los nodos se lanzan como **LifecycleNode**. En `J8_launch.py` se disparan
transiciones `CONFIGURE` y `ACTIVATE` con un retardo (TimerAction). Este comportamiento
es importante si algunos nodos dependen de TF, servicios o tópicos que tardan en aparecer.
